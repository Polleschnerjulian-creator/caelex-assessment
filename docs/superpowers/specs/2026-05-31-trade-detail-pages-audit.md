# Trade UI Redesign — Phase 3E Scope: Detail/Edit Pages Audit

**Date:** 2026-05-31
**Status:** Audit (read-only) — backlog for a future Phase 3E
**Branch:** `fix/trade-to-92`
**Predecessors:** Phase 1 (nav/home/tokens), Phase 2 (search-pickers + datasheet auto-classify), Phase 3A (shared `TradeTable` + list migration — items/operations/parties)
**Parallel WIP:** 3B (licenses), 3C (screening triage), 3D (documents) are being designed concurrently.

---

## Purpose

Phase 3A modernised the three **list** pages onto a shared `TradeTable<T>`. This audit maps the **detail/edit** surfaces (and any remaining rough UI) and proposes a prioritized **3E backlog**, measured against the Trade north star:

> **Modern · scannable · maximally guides the user · automates where possible — at ~zero external runtime cost.**

Scope explored: every page under `src/app/(trade)/trade/**`. All paths below are absolute-from-repo-root under that tree.

## Leverage — do this one first

**Item 2 (in-context Astra on detail pages) unlocks the most**, with **Item 1 (wizard re-theme) as the trivial-cost prerequisite warm-up.**

The north star's centre of gravity is _"maximally guides the user / automates."_ Trade already has the hard parts built — a tool-using Astra engine, an evidence-grade classification copilot, a risk engine, a sanctions-screening engine. The failure is **integration**: all of that intelligence is reachable only from standalone pages, never from the record where the user is actually stuck making a regulated decision. Dropping a contextual Astra affordance onto the three detail pages (reusing the _existing_ `prefill` deep-link pattern, so it's M-effort, not L) converts every catch-all banner, every potential-match triage, and every blank ECCN field into a one-click "explain/assist" moment. It also establishes the pattern that Items 4 (classify-from-item) and 7 (false-positive assist) extend.

Item 1 ships in the same PR window because it's a pure token swap on 2 files (22 legacy occurrences) and removes the most embarrassing visual seam in the product — the guided wizard that looks like it belongs to a different app.

Items 3/5/7 (TradeTable + jump-nav + inline-edit) are the **scannability** follow-through that reuses the 3A primitive across the detail pages, and Item 6 should be **explicitly deferred to land on top of 3B** rather than competing with it.

## Notes for whoever specs 3E

- **Do not rebuild what 3B/3C/3D own.** Licences detail (Item 6) sits on 3B; the doc sub-surface long-tail (7e) is downstream of 3D's decisions. This audit's _core_ 3E = Items 1–5 + 7–8 (the item/operation/party/home surfaces 3A–3D don't cover).
- **Zero new runtime deps** holds — every item reuses an existing primitive (`TradeTable`, `EmptyStateRich`/`prefill`, `FieldRow`, classify server actions, Astra).
- **Cheapest-first ordering** is deliberate: Item 1 (S) and Item 2 (M) deliver the biggest north-star jump for the least code and risk.
