ATLAS — System Architecture

How should ATLAS integrate into the Caelex ecosystem? Pick the architecture that fits.

A

Embedded App Section

Route: /atlas/* as new top-level section alongside /dashboard/*

Layout: Own sidebar + layout, shares root layout with Caelex

Data: Direct Prisma access to same DB, shared models

Auth: Same NextAuth session, new "atlas" permission scope

✓ Fastest to build · ✓ Shared billing · ✓ Reuses all existing data
✗ Tighter coupling · ✗ Harder to sell separately later

B

Separate Next.js App (same repo, Turborepo)

Route: atlas.caelex.eu as own subdomain

Layout: Completely independent app with own design system

Data: Shared Prisma package, same DB but own models

Auth: Shared auth package, separate session handling

✓ Clean separation · ✓ Independent deploys · ✓ Sellable standalone
✗ Monorepo overhead · ✗ More setup · ✗ Duplicate infra code

C

Hybrid — Embedded with Feature Boundary

Route: /atlas/* with own route group (atlas)

Layout: Own layout.tsx + sidebar, Bloomberg-style dark theme

Data: New Prisma models prefixed Atlas*, reads from existing models

Auth: Same session, gated by atlas:read / atlas:write permissions + separate API key tier

✓ Clean boundaries within one codebase · ✓ Shared auth & billing · ✓ Can extract to monorepo later
✗ Needs discipline to maintain boundaries · Recommended approach
